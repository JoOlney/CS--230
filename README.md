# CS--230
Operating Platform: SNHU 2026

The Gaming Room is the client for this project. They currently offer a game called Draw It or Lose It — a web-based game modeled after the 1980s television program Win, Lose or Draw, in which teams compete to identify images rendered progressively from a stock drawing library. The game was originally built for Android only, and The Gaming Room wanted it expanded into a distributed, cross-platform web application accessible from any modern browser on Linux, macOS, Windows, iOS, and Android devices. They needed a design that supported multiple simultaneous games, teams, and players — with unique names enforced at every level — and they required that only one instance of the game service exist in memory at any time. The design had to minimize licensing costs, support cloud deployment, and secure all API endpoints. My role was to produce a complete software design document covering the domain model, platform evaluation, and architectural recommendations to guide the development team.

I believe I did particularly well in documenting the design patterns and explaining their purpose clearly enough for both a technical development team and a non-technical client to understand. The Singleton pattern applied to GameService and the Iterator pattern used throughout Game, Team, and Player were described not just in terms of how they work in code, but why they were chosen — what specific requirement each pattern addresses and what would go wrong without it. I also did well in the platform evaluation section, where I assessed Linux, macOS, Windows, and mobile platforms across server-side hosting, client-side delivery, and development tooling in a structured, side-by-side format that gave the client a clear basis for decision-making rather than just a list of features.

Working through the design document first forced me to fully think through the class hierarchy and the relationships between objects before writing any code. Defining the Entity base class, then deriving Game, Team, and Player from it, clarified exactly what shared behavior could be inherited versus what needed to be implemented separately at each level. The domain model section in particular was useful because it required me to articulate the Iterator pattern's role in enforcing unique names.

If I could revise one part, I would expand the System Architecture View section. The template notes that this section is not required for these specific projects, so I left it minimal. However, in a real-world delivery this is one of the most critical sections — it is where the physical deployment topology is described, including how the web server, application server, database, and CDN relate to one another. I would improve it by adding a labeled architecture diagram showing the three-tier structure (presentation, application, data), the cloud infrastructure components (load balancer, application server instances, managed database), and the network boundaries between them. This would make the document more complete and significantly more useful to a development or DevOps team picking up the project.

I interpreted the client's needs by working from their stated goals outward into technical decisions. The requirement that the game be accessible on any platform directly drove the choice of a responsive HTML5/JavaScript front-end over native apps. The requirement that only one game service instance exist in memory directly drove the Singleton pattern implementation. The requirement that names be unique across concurrent users drove both the Iterator pattern in the prototype and the recommendation for database-level unique constraints in production. Considering user needs is essential because a technically elegant system that solves the wrong problem is worthless. Every design decision needs a traceable reason rooted in what the client actually needs — otherwise developers make assumptions that compound into a product that has to be rebuilt. User-centered design also reduces rework: the earlier needs are understood and locked into the design, the less expensive it is to accommodate them.

My approach was to move from requirements to constraints to design patterns to architecture — in that order, without skipping steps. I started by clearly separating functional requirements (what the software must do) from non-functional requirements (how well it must do it), because these two categories drive different types of design decisions. From there I identified the constraints — statelessness, singleton enforcement, concurrent name uniqueness, cross-platform compatibility — that bounded the solution space. Only then did I look for design patterns that solved those constraints cleanly. In the future I would add two things to this process. First, I would create a formal requirements traceability matrix to explicitly link every requirement to the design element that addresses it, so nothing gets lost between documents. Second, I would prototype the highest-risk components.
